# RankRoom Judge0 Implementation: Fixes & Improvements Plan

## Overview

This plan addresses 20 identified issues in the RankRoom Judge0 implementation, prioritized by impact on correctness, scalability, and security. The fixes are organized into phases for safe, incremental deployment.

---

## Phase 1: Critical Correctness Fixes

### 1.1 Unify TypeScript Execution Behavior

**Issue**: TS transpilation in `execution.service.ts` is inconsistent with Judge0 TS language ID 74 usage.

**Current State**:

- `execution.service.ts:164` transpiles TS→JS for wrapped mode
- `judge.service.ts` still supports language ID 74 for raw mode
- Users get different behavior between `/run` and `/submit`

**Fix**:

1. Add configuration flag `TS_EXECUTION_MODE` (default: `transpile`)
2. Make TS handling consistent across all execution paths
3. Update `execution.service.ts` to use the same transpilation logic for both wrapped and raw modes
4. Document the behavior clearly

**Files to modify**:

- `apps/api/src/config/execution.ts` - Add TS execution mode config
- `apps/api/src/services/execution.service.ts` - Unify transpilation
- `apps/api/src/services/judge.service.ts` - Remove direct TS language ID usage when transpilation enabled
- `apps/api/src/routes/problems.ts` - Add transpilation info to response

**Pseudo-code**:

```typescript
// config/execution.ts
export const TS_EXECUTION_MODE =
  process.env["TS_EXECUTION_MODE"] ?? "transpile"; // "transpile" | "native"

// services/execution.service.ts
const isTypeScript = language === "typescript";
const effectiveLanguage =
  isTypeScript && TS_EXECUTION_MODE === "transpile" ? "javascript" : language;
const effectiveUserCode =
  isTypeScript && TS_EXECUTION_MODE === "transpile"
    ? ts.transpileModule(userCode, {
        /* config */
      }).outputText
    : userCode;
```

**Issue**: `serializeInput()` can be lossy for edge cases.

**Fix**:

2. Handle big integers with string representation
3. Add explicit null/undefined handling
4. Add validation for nested structures

**Files to modify**:

- `apps/api/src/services/input-serializer.service.ts`

**Pseudo-code**:

```typescript
function serializeString(value: string): string {
  // Escape newlines, backslashes, and control characters
  return value
    .replace(/\\/g, "\\\\")
    .replace(/\n/g, "\\n")
    .replace(/\r/g, "\\r")
    .replace(/\t/g, "\\t");
}

function serializeBigInt(value: bigint): string {
  // Convert to string to avoid precision loss
  return value.toString();
}
```

### 3.2 Improve Wrapper Detection

**Issue**: Marker-based detection can false-positive.

**Fix**:

1. Use more specific markers (function signatures, not variable names)
2. Add AST-based detection for critical languages
3. Add explicit "already wrapped" flag in API

**Files to modify**:

- `apps/api/src/config/templates.ts` - Improve `isRankRoomWrappedSubmission()`

**Better detection**:

```typescript
export function isRankRoomWrappedSubmission(sourceCode: string): boolean {
  // Look for specific wrapper patterns, not just variable names
  const wrapperPatterns = [
    /require\s*\(\s*['"]fs['"]\s*\)\s*\.readFileSync\s*\(\s*0/, // JS wrapper
    /sys\.stdin\.read\(\)\.strip\(\)\.split/, // Python wrapper
    /public\s+class\s+Main\s*\{/, // Java wrapper
    /int\s+main\s*\(\s*\)\s*\{/, // C/C++ wrapper with specific signature
  ];

  return wrapperPatterns.some((pattern) => pattern.test(sourceCode));
}
```

### 4.4 Add Retry Idempotency

**Issue**: Retrying on transient failures can create duplicate Judge0 submissions if the original POST succeeded but response was lost.

**Fix**:

1. Store submission token before polling to detect duplicates
2. Add idempotency key to Judge0 requests where supported
3. Implement deduplication logic for retry scenarios
4. Add logging to track retry attempts and outcomes

**Files to modify**:

- `apps/api/src/services/judge.service.ts` - Add idempotency tracking
- `apps/api/src/jobs/submissionWorker.ts` - Add duplicate detection

**Pseudo-code**:

```typescript
// Track tokens to prevent duplicate submissions
const submissionTokens = new Map<string, string>();

async function createSubmissionWithIdempotency(input) {
  const idempotencyKey = generateIdempotencyKey(input);

  if (submissionTokens.has(idempotencyKey)) {
    return submissionTokens.get(idempotencyKey);
  }

  const token = await createSubmission(input);
  submissionTokens.set(idempotencyKey, token);
  return token;
}
```

  1.1, 1.2, 1.3          2.1, 2.2, 2.3          3.1, 3.2, 3.3        4.1, 4.2, 4.3        5.1, 5.2

## Risk Assessment

| Change                  | Risk                             | Mitigation                         |
| ----------------------- | -------------------------------- | ---------------------------------- |
| TS transpilation change | Breaking existing TS submissions | Feature flag with rollback         |
| Payload limits          | Reject valid submissions         | Generous defaults, gradual rollout |
| Contest strictness      | Break existing contests          | Grace period with warnings         |
| Wrapper changes         | Break edge cases                 | Comprehensive test suite           |
| Rate limiting           | Slow down legitimate users       | Per-user limits, not global        |

---

## Success Metrics

1. **Consistency**: Same verdict for same code across all execution paths
2. **Performance**: <5s average submission latency at 50 concurrent users
3. **Reliability**: <1% submission failure rate due to infrastructure
4. **Security**: Zero successful sandbox escapes or resource exhaustion
5. **Scalability**: Support 100 concurrent submissions without degradation
